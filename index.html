DRIVEU — Control Flow Reference
Complete step-by-step flow for Customer · Driver · Admin — mapped to your PostgreSQL schema

Decision / branch point

Authentication
01
Open App — Enter Phone Number
Customer enters their 10-digit mobile number. The app validates the format client-side before making a server call.
READ user_phones (check existence)
02
OTP Sent to Phone
A 6-digit OTP is generated and sent via SMS. A record is created in otp_requests with an expiry timestamp and a counter.
INSERT otp_requests phone_number, otp_code, expires_at, max_attempts=5
ℹ️ Customer has 5 attempts max. Each wrong entry increments attempt_count. If attempt_count ≥ max_attempts, OTP is locked.
03
Customer Enters OTP
Customer enters the 6-digit code. System checks otp_code, expires_at, is_used, and attempt_count.
UPDATE otp_requests → is_used=true
✓ Valid OTP
Mark is_used = true. Proceed to step 04.
✗ Wrong / Expired
Increment attempt_count. Show error. If max attempts hit → lock OTP → customer must request a new one (back to step 02).
04
New or Returning Customer?
⑆
Check if a users record with this phone exists in user_phones.
NEW
Create users (id=UUID, status_id=active) → Create user_phones (is_primary=true, is_verified=true) → Create customer_profiles (name, email) → Assign CUSTOMER role via user_roles → Land on Home Screen.
EXISTING
Validate user_statuses (must be active). If blocked → show error. If active → land on Home Screen.
INSERT users INSERT user_phones INSERT customer_profiles INSERT user_roles
Booking Creation
05
Home Screen — Select Trip Type
Customer picks the kind of trip they want (e.g., Hourly, Outstation, Airport Drop). List is fetched from trip_types.
READ trip_types
06
Select Vehicle Type & Transmission
Customer selects car category (Sedan, SUV, etc.) and transmission preference (Auto / Manual). These determine which drivers are eligible and which pricing config applies.
READ vehicle_types READ transmission_types
07
Enter Pickup Location
Customer searches for pickup address via Google Places. A locations record is created storing address, lat, lng, and Google place_id.
INSERT locations (pickup)
08
Enter Destination (Optional)
For outstation / airport trips, a destination is required. For hourly bookings, it may be skipped — destination_location_id remains NULL.
INSERT locations (destination) — optional
09
Select Date, Time & Duration
Customer picks scheduled_at (start time). For hourly trips, they also set booked_duration_hours. System checks if the requested slot is in the future.
10
Fare Estimate Calculated & Displayed
System queries pricing_configs matching city_id + trip_type_id + vehicle_type_id. It then applies tiers from pricing_tiers based on booked_duration_hours. If surge is active (surge_active_from ≤ now ≤ surge_active_to), the surge_multiplier is applied.
READ pricing_configs READ pricing_tiers
Base fare pulled from pricing_configs.base_fare
Duration fare = tier rate × hours from pricing_tiers
Platform fee + tax added on top
Total fare shown to customer before confirmation
11
Customer Reviews & Confirms Booking
Customer sees full booking summary: trip type, vehicle, pickup, destination, scheduled time, fare breakdown. They tap "Confirm Booking".
INSERT bookings → status: SEARCHING idempotency_key used to prevent duplicate submissions
STATUS → SEARCHING
⚠️ The idempotency_key on the bookings table ensures that if the customer taps "Confirm" multiple times (network issues), only one booking is ever created.
Driver Matching & Trip Lifecycle
12
System Searches for Available Driver
The system finds online drivers in the same city (driver_profiles.city_id) who serve the pickup zone (driver_zones), support the required vehicle type (driver_vehicle_types) and transmission (driver_transmission_types), and are not already booked (driver_availability).
READ driver_profiles (is_online=true) READ driver_locations (proximity) READ driver_availability (no conflict) INSERT booking_requests → status: PENDING
ℹ️ The system sends a request to the nearest available eligible driver. The driver must manually accept or reject — there is no auto-expiry timer. The request stays PENDING until the driver responds.
13
Waiting for Driver Response
Customer sees a "Searching for driver…" screen. The booking remains in SEARCHING status while a booking_request is PENDING with a driver.
ACCEPTED
booking_requests → status: ACCEPTED · bookings → status: CONFIRMED · Proceed to step 14.
REJECTED
booking_requests → status: REJECTED · System finds the next nearest available driver and sends a new booking_requests record (status: PENDING) to them. Loop repeats.
NO DRIVER FOUND
If no eligible drivers exist or all have rejected → bookings → status: NO_DRIVER_FOUND · Customer is notified and can try again or modify their booking details.
UPDATE booking_requests → status UPDATE bookings → status
14
Booking Confirmed — Driver Details Shown
Customer sees driver name, photo, average rating, vehicle info, and live ETA. The booking is confirmed.
READ driver_profiles, driver_locations
STATUS → CONFIRMED
⛔ Cancellation Window (Steps 11–17)
Customer can cancel at any point before OTP verification. Tapping "Cancel" creates a cancellation record (reason, cancelled_by=customer user_id, booking_id), then updates bookings → status: CANCELLED, cancelled_at=now().
15
Driver En Route — Live Tracking
Customer sees driver's real-time location on a map. Location data is continuously updated in driver_locations by the driver's app.
READ driver_locations (polling / websocket)
16
Driver Arrives at Pickup
Driver marks themselves as arrived. A 6-digit OTP is generated and saved to booking_otps. The driver shows this OTP to the customer verbally or on-screen.
INSERT booking_otps (otp_code, is_verified=false)
17
Customer Verifies OTP — Trip Starts
Customer enters the OTP on their app. System validates it against booking_otps.otp_code. On match, OTP is marked verified and the trip begins.
UPDATE booking_otps → is_verified=true, verified_at=now() UPDATE bookings → status: ONGOING INSERT driver_availability → booked_from=now()
STATUS → ONGOING
🔒 Once OTP is verified, cancellation is no longer possible from the customer side.
18
Trip in Progress
Customer is in the vehicle. Trip duration is tracked. If the trip exceeds booked_duration_hours, overtime is calculated using the overtime rate from pricing_tiers.
Overtime hours = actual_duration − booked_duration_hours
Overtime fare stored in bookings.overtime_fare
Total fare updated = base_fare + duration_fare + overtime_fare + tax + platform_fee − discount
19
Trip Ends — Driver Marks Complete
Driver taps "End Trip". Booking status is updated to COMPLETED. Driver's availability window is closed. Earnings are credited.
UPDATE bookings → status: COMPLETED, drop_at=now() UPDATE driver_availability → booked_till=now() UPDATE driver_profiles → total_trips++, total_earnings += fare
STATUS → COMPLETED
Payment & Post-Trip
20
Payment — Cash or UPI
Customer pays the total fare. A booking_payments record is created with the chosen method.
INSERT booking_payments payment_method: CASH or UPI payment_status: SUCCESS / FAILED / PENDING
CASH
Customer hands cash to driver. App marks payment_status = SUCCESS immediately.
UPI
UPI payment flow initiated. On success → payment_status = SUCCESS, paid_at = now(). On failure → status = FAILED, customer can retry.
21
Rate the Driver
Customer gives 1–5 stars and optional comments. A ratings record is created. The driver's avg_rating in driver_profiles is recalculated.
INSERT ratings (booking_id, customer_id, driver_id, stars, comments) UPDATE driver_profiles.avg_rating (recalculated)
ℹ️ Rating is optional but shown as a prompt. UNIQUE constraint on (booking_id, customer_id) ensures one rating per trip.
22
Raise a Support Ticket (Optional)
Customer can raise a support_tickets record any time — during or after a booking. They choose a category, describe the issue, and optionally attach a screenshot.
INSERT support_tickets → status: OPEN category: PAYMENT_ISSUE / DRIVER_ISSUE / BOOKING_ISSUE / APP_ISSUE / OTHER

Onboarding & Registration
01
Open App — Phone OTP Authentication
Same OTP flow as customer. Driver enters phone number → OTP sent (INSERT otp_requests) → Driver enters code → verified.
INSERT otp_requests UPDATE otp_requests → is_used=true
02
New Driver Registration — Profile Creation
If no user exists for this phone, the app creates a new driver account. Driver selects their city from the available cities list.
INSERT users (status_id=active) INSERT user_phones (is_primary=true, is_verified=true) INSERT driver_profiles (city_id, is_online=false, is_verified=false) INSERT user_roles (DRIVER role)
⚠️ is_verified = false by default. Driver cannot go online until KYC is approved by admin.
03
Select Vehicle Types & Transmission
Driver selects which types of vehicles they can drive (Sedan, SUV, etc.) and which transmissions they handle (Automatic, Manual). These determine which bookings they'll be eligible for.
INSERT driver_vehicle_types (driver_id, vehicle_type_id) — for each selection INSERT driver_transmission_types (driver_id, transmission_type_id)
04
Select Service Zones
Driver picks which city zones they're willing to serve. Only bookings with pickup locations in these zones will be offered to them.
INSERT driver_zones (driver_id, zone_id) — for each zone selected
05
Upload KYC Documents
Driver uploads the required identity and licensing documents. Each document is stored in driver_documents with its type and side.
Aadhar Card — FRONT side
Aadhar Card — BACK side
Driving License — FRONT side
Driving License — BACK side
PAN Card — FULL
Profile Photo — FULL (stored as PROFILE_IMAGE type)
INSERT driver_documents × 6 (document_type, document_side, file_key, file_provider='S3', is_verified=false)
06
KYC Submitted — Awaiting Admin Review
Driver submits for KYC review. A log entry is created marking the review as pending.
INSERT driver_kyc_logs → status: pending
KYC STATUS → pending
APPROVED by Admin
driver_kyc_logs → status: approved, reviewed_by=admin_user_id, reviewed_at=now() · driver_profiles.is_verified = true · Driver can now subscribe to a plan. Proceed to step 07.
REJECTED by Admin
driver_kyc_logs → status: rejected, reason=text, reviewed_by, reviewed_at · driver_profiles.kyc_reapply_after = future timestamp · Driver sees rejection reason. They may re-upload corrected documents → INSERT new driver_documents → INSERT driver_kyc_logs → status: resubmitted → back to admin review queue.
UPDATE driver_kyc_logs UPDATE driver_profiles.is_verified UPDATE driver_profiles.kyc_reapply_after (on rejection)
Subscription & Going Online
07
Browse Subscription Plans
KYC-approved driver sees available plans for their city. Each plan shows price, duration (days), and a list of features from plan_features.
READ subscription_plans (city_id match, is_active=true) READ plan_features + features (for feature list)
08
Select Plan & Initiate Payment
Driver selects a plan and proceeds to pay via Razorpay. A payment record is created immediately in CREATED status, along with a subscription in PENDING status.
INSERT driver_subscriptions → status: PENDING, payment_id=NULL initially INSERT payments → status: CREATED, razorpay_order_id set
PAYMENT SUCCESS
payments → status: SUCCESS, razorpay_payment_id set, paid_at=now() · driver_subscriptions → status: ACTIVE, payment_id linked, starts_at=now(), expires_at=now()+duration_days.
PAYMENT FAILED
payments → status: FAILED · driver_subscriptions remains PENDING · Driver can retry payment. A new payments record is created on retry.
UPDATE payments → status: SUCCESS/FAILED UPDATE driver_subscriptions → status: ACTIVE, payment_id
SUBSCRIPTION → ACTIVE
ℹ️ The circular FK between payments and driver_subscriptions is resolved by: first creating the subscription with payment_id=NULL, then creating the payment with subscription_id set, then updating the subscription with payment_id.
09
Go Online
Driver taps "Go Online". System verifies: (1) is_verified=true, (2) active subscription exists, (3) subscription not expired. If all pass, driver goes online and starts sharing location.
UPDATE driver_profiles → is_online=true INSERT/UPDATE driver_locations → lat, lng, is_online=true, updated_at=now()
NOT VERIFIED
Blocked. Show "KYC approval pending" message.
NO ACTIVE SUBSCRIPTION
Blocked. Redirect to plan selection (step 07). If subscription expired → status: EXPIRED → prompt renewal.
Receiving & Managing Booking Requests
10
Booking Request Arrives
System pushes a booking request to the driver via notification/WebSocket. Driver sees: pickup address, zone, trip type, vehicle type, duration, and estimated fare. The booking_requests record is in PENDING status.
READ booking_requests (PENDING, driver_id=self)
REQUEST STATUS → PENDING
🔴 IMPORTANT: There is NO auto-expiry on booking requests. The driver must manually tap Accept or Reject. The request will remain PENDING until the driver responds.
11
Driver Reviews Request — Accepts or Rejects
Driver sees full booking details on screen and makes a decision.
✓ ACCEPT
booking_requests → status: ACCEPTED, responded_at=now() · bookings → status: CONFIRMED, driver_id=this driver · Customer is notified of driver name, photo, ETA. Driver receives pickup navigation.
✗ REJECT
booking_requests → status: REJECTED, responded_at=now() · Booking remains SEARCHING · System creates a new booking_requests for the next available eligible driver. This driver is skipped for this booking.
UPDATE booking_requests → status, responded_at UPDATE bookings → status: CONFIRMED, driver_id (on accept)
12
Navigate to Customer — Share Live Location
After accepting, driver navigates to the pickup point. The driver app continuously updates driver_locations so the customer can see real-time ETA on their screen.
UPDATE driver_locations → lat, lng, updated_at (continuous updates)
13
Arrive at Pickup — Generate OTP
Driver taps "I've Arrived". The system generates a 6-digit OTP and stores it in booking_otps. Driver verbally tells the OTP to the customer or shows their screen.
INSERT booking_otps (otp_code, is_verified=false)
14
Customer Enters OTP — Trip Starts
Customer enters OTP on their app. On match, the trip begins. Driver's availability is blocked to prevent double-booking.
UPDATE booking_otps → is_verified=true, verified_at=now() UPDATE bookings → status: ONGOING INSERT driver_availability → booked_from=now(), driver_id
BOOKING → ONGOING
Trip Completion & Earnings
15
End Trip — Mark Complete
Driver taps "End Trip" at the drop point. The booking is marked COMPLETED. Driver availability window is closed, and earnings are tallied.
UPDATE bookings → status: COMPLETED, drop_at=now(), total_fare (final with overtime) UPDATE driver_availability → booked_till=now() UPDATE driver_profiles → total_trips++, total_earnings += total_fare
BOOKING → COMPLETED
16
Collect Payment — Customer Pays
Driver collects fare (CASH handed to driver, or confirms UPI payment received). The booking_payments record is updated to SUCCESS.
UPDATE booking_payments → payment_status: SUCCESS, paid_at=now()
17
Rating Received from Customer
Customer rates the driver (1-5 stars). The driver's avg_rating is updated automatically.
UPDATE driver_profiles.avg_rating (recalculated from all ratings)
18
Go Offline (Optional)
Driver can toggle offline at any time when not in a trip. No new requests will be sent while offline.
UPDATE driver_profiles → is_online=false UPDATE driver_locations → is_online=false
19
Subscription Renewal
When driver_subscriptions.expires_at passes, subscription status moves to EXPIRED. Driver is blocked from going online and is prompted to renew (repeat steps 07–08).
UPDATE driver_subscriptions → status: EXPIRED
SUBSCRIPTION → EXPIRED
⚠️ Driver is immediately unable to go online once subscription expires. They must purchase a new plan to resume accepting bookings.

Admin Authentication
01
Admin Login
Admin logs in via secure credentials. System verifies admin_profiles (is_verified=true) and checks that the associated user_roles contains an ADMIN role with sufficient permissions via role_permissions.
READ admin_profiles (email lookup) READ user_roles → roles → role_permissions → permissions UPDATE admin_profiles → last_login=now() INSERT audit_logs (login event)
KYC Management — Driver Verification
02
View KYC Review Queue
Admin sees a list of all drivers with driver_kyc_logs.status = 'pending' or 'resubmitted'. Sorted by submission date.
READ driver_kyc_logs WHERE status IN ('pending','resubmitted')
03
Open Driver Profile — Review Documents
Admin opens a driver's submission. They see all uploaded documents from driver_documents — each document rendered from the file_key (S3 or Cloudinary). Admin reviews each one.
READ driver_documents WHERE driver_id=X READ driver_profiles (city, zones, vehicle types, transmission types)
Verify Aadhar front + back match same person
Verify Driving License validity and type matches selected vehicle_types
Verify PAN card details
Confirm profile photo is a clear selfie
04
Approve or Reject KYC
✓ APPROVE
INSERT driver_kyc_logs → status: approved, reviewed_by=admin_uuid, reviewed_at=now() · UPDATE driver_profiles → is_verified=true · UPDATE driver_documents → is_verified=true (for each doc) · Driver receives push notification: "KYC Approved! Subscribe to a plan to go online."
✗ REJECT
Admin selects reason (e.g., "Blurry Aadhar back", "Driving license expired") · INSERT driver_kyc_logs → status: rejected, reason=text, reviewed_by, reviewed_at · UPDATE driver_profiles → kyc_reapply_after = now() + cooldown_period · Driver notified with rejection reason and resubmission instructions.
INSERT driver_kyc_logs UPDATE driver_profiles.is_verified / .kyc_reapply_after UPDATE driver_documents.is_verified (on approval) INSERT audit_logs
Geography Management
05
Manage States
Admin adds or edits state records. States are the top-level geography unit.
INSERT / UPDATE states (name, created_by, updated_by)
06
Manage Districts
Admin adds districts under a state. state_id is required.
INSERT / UPDATE districts (state_id, name)
07
Manage Cities
Admin adds cities under a district. Cities are the operational unit — all pricing, subscriptions, drivers, and bookings are city-scoped.
INSERT / UPDATE cities (district_id, name)
08
Manage City Zones
Admin defines zones within a city (e.g., "North Zone", "Airport Zone"). Zones control which drivers serve which pickup areas.
INSERT / UPDATE city_zones (city_id, name)
Pricing Configuration
09
Create Pricing Config
Admin creates a pricing rule for a specific combination of city + trip type + vehicle type. The UNIQUE constraint ensures no duplicate configs for the same combination + effective date.
INSERT pricing_configs city_id + trip_type_id + vehicle_type_id + effective_from (unique combo) base_fare, min_hours, max_hours, surge_multiplier
10
Add Pricing Tiers to Config
For each pricing config, admin adds hour-range tiers. E.g., hours 0–4 at ₹150/hr, hours 4–8 at ₹120/hr. Ranges cannot overlap (UNIQUE on config_id + from_hour + to_hour).
INSERT pricing_tiers (config_id, from_hour, to_hour, rate_per_hour, label, sort_order)
CHECK: to_hour > from_hour enforced by DB constraint
UNIQUE: (config_id, from_hour, to_hour) — no overlapping ranges
11
Activate Surge Pricing
Admin sets surge window on a pricing config. During this window, all fares for that config are multiplied by surge_multiplier.
UPDATE pricing_configs → surge_multiplier, surge_active_from, surge_active_to
⚠️ DB CHECK constraint enforces: surge_active_to IS NULL OR surge_active_to > surge_active_from. Surge ≥ 1.0 (never discounts).
Subscription Plan Management
12
Create Subscription Plan
Admin creates a driver subscription plan for a city. Plan has a name, price, and validity in days.
INSERT subscription_plans (city_id, name, price, duration_days, is_active=true)
13
Add Features to Plan
Admin maps features (from the features master table) to the plan. Each feature gets a value (e.g., "Unlimited rides", "10 bookings/day", "Priority support").
INSERT features (code, name, description_text) — if new feature INSERT plan_features (plan_id, feature_id, value)
ℹ️ UNIQUE constraint on (plan_id, feature_id) ensures no duplicate feature per plan.
14
Activate or Deactivate Plan
Admin can toggle a plan's visibility. Deactivated plans (is_active=false) are hidden from drivers but existing active subscriptions continue until expiry.
UPDATE subscription_plans → is_active = true/false
User Management
15
View & Search All Users
Admin can search users (customers, drivers, admins) by phone, name, or email. Results show status, roles, and profile type.
READ users + user_phones + customer_profiles / driver_profiles READ user_roles + roles (to show assigned roles)
16
Block or Activate a User
Admin changes a user's status. Blocked users cannot log in. Active users can use the platform normally.
UPDATE users → status_id (lookup user_statuses table for appropriate status code)
BLOCK USER
Set status_id to "blocked" status. If the user is a driver currently online, they are forced offline: UPDATE driver_profiles.is_online = false, UPDATE driver_locations.is_online = false.
ACTIVATE USER
Set status_id to "active" status. User can log in immediately.
17
Assign or Remove Roles
Admin assigns roles (CUSTOMER, DRIVER, ADMIN, SUPPORT_AGENT, etc.) to users. A UNIQUE constraint prevents duplicate role assignments.
INSERT user_roles (user_id, role_id, assigned_at) DELETE user_roles (to remove a role)
Booking Management
18
View All Bookings
Admin views a full list of bookings with rich filters: city, status, trip type, vehicle type, date range, customer or driver name.
READ bookings + customer_profiles + driver_profiles + trip_types + vehicle_types + cities
SEARCHING CONFIRMED ONGOING COMPLETED CANCELLED
19
View Booking Detail
Admin opens a booking to see the complete picture: fare breakdown, driver request history, OTP verification log, payment status, cancellation details, and rating.
READ bookings (all fare fields) READ booking_requests (full driver request history for this booking) READ booking_otps READ booking_payments READ cancellation (if cancelled) READ ratings (if rated) READ locations (pickup + destination)
Support Ticket Management
20
View Support Queue
Admin/support agent sees tickets filterable by status, category, city, and date. Default view shows OPEN tickets sorted by creation time (oldest first).
READ support_tickets WHERE status='OPEN' ORDER BY created_at ASC
21
Open Ticket — Investigate
Agent opens a ticket. They see the customer's description, screenshot (if any), and the linked booking (if relevant). They can review the booking's full timeline to investigate the issue.
READ support_tickets + customer_profiles READ bookings (via booking_id FK, if attached)
22
Update Ticket Status
Agent moves the ticket through the resolution workflow.
OPEN → IN_PROGRESS (agent starts working on it)
IN_PROGRESS → RESOLVED (issue resolved, resolved_by + resolved_at set)
RESOLVED → CLOSED (customer acknowledged or auto-closed after 48h)
UPDATE support_tickets → status, resolved_by=admin_uuid, resolved_at=now() INSERT audit_logs
Roles & Permissions Management
23
Create & Manage Roles
Admin creates named roles (e.g., SUPER_ADMIN, SUPPORT_AGENT, CITY_MANAGER). Each role is described and assigned a set of permissions.
INSERT roles (name, description)
24
Create Permissions & Assign to Roles
Admin creates named permissions (e.g., APPROVE_KYC, VIEW_BOOKINGS, MANAGE_PRICING, RESOLVE_TICKETS). Then maps them to roles via role_permissions.
INSERT permissions (name, description) INSERT role_permissions (role_id, permission_id) DELETE role_permissions (to revoke)
Audit & Monitoring
25
View Audit Logs
Admin can view every API call made to the system. Each log entry shows who called which endpoint, their IP, request payload, response status, and how long it took. Useful for debugging, security monitoring, and compliance.
READ audit_logs Filters: user_id, api_route, response_status, ip_address, date range
All KYC decisions written to audit_logs (reviewed_by, action, timestamp)
All pricing changes logged
All user status changes logged
All role/permission changes logged
Failed API calls (4xx/5xx) surfaced with error_message
26
View Driver Subscription & Payment Reports
Admin reviews active subscriptions, expiring soon subscriptions, and all payment transactions. Can identify failed payment patterns or subscription churn by city.
READ driver_subscriptions (status, expires_at) READ payments (status, amount, razorpay_payment_id) JOIN driver_profiles + cities
